Issue Type: Bug Report
Recommended Visibility: Public issue
Suggested Title: Persist demo time-off submissions so follow-up history lookups reflect the request

## Description
`request_my_time_off()` returns a success payload, but it does not update any in-memory or file-backed history state. As a result, a user can "submit" a request and then immediately ask for their time-off history without seeing the request they just created.

## Component
- [ ] Chat Connector
- [ ] Flowise Flow
- [x] Demo MCP Server
- [ ] Documentation
- [ ] Other

## Steps to Reproduce
1. Call `request_my_time_off(start_date="2026-05-20", end_date="2026-05-21", time_off_type="vacation")`.
2. Then call `get_current_user_time_off_history()`.
3. Observe that the new request is missing from the returned history.

## Expected Behavior
The demo tool should either persist the new request in demo state or clearly state that write operations are stateless/no-op simulations.

## Actual Behavior
The tool reports `"status": "submitted"` but does not affect subsequent reads.
