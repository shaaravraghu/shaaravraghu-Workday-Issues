Issue Raising Procedure for base.build

Purpose
Prepare and raise the issue drafts in this folder using the repository's contribution norms and enough evidence for maintainers to triage quickly.

Suggested workflow
1. Read `issue-drafts/01-issue-inventory.txt` to decide which issues to raise first.
2. Start with the highest-risk items:
   security, correctness, data loss, and broken public API behavior.
3. Confirm whether an issue already exists in GitHub before opening a duplicate.
4. Copy the corresponding `.txt` file into a new GitHub issue.
5. Keep the title, summary, affected files, expected result, actual result, and acceptance criteria intact.
6. Add repo-specific labels such as `bug`, `documentation`, `security`, `performance`, or `enhancement`.
7. If the issue touches public API behavior, mention whether a fix is expected to be source-compatible and behavior-compatible.
8. If the issue was identified by source inspection rather than an executed test, keep the `Verification status` field so maintainers know how it was validated.
9. For larger issues, attach a minimal proof-of-concept test or code snippet when you raise the issue.
10. Comment on the issue if you plan to work on it so maintainers can coordinate.

Contributor etiquette
Create or identify an issue before starting work.
Ask whether the issue is still relevant if there has been no recent activity.
State clearly if you want to work on it.
Ask follow-up questions when behavior is unclear.
If the issue affects UI, UX, or docs structure, ask a maintainer or designer for review early.
